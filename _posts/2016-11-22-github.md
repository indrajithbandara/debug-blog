#### Comofaz pra voltar? 
O $git reset tem 3 funçoes, o Soft, o Mixed e o Hard. 

$ --soft: vai pegar as modificacoes e vai matar o commit mas o arquivo vai estar em stage com a modificacao proto para ser comitado de novo
$ --mixed:  vai matar o comit também só que ele vai voltar os arquivos para antes do stage
$ --reset ignorar a existencia do commit e vai matar tudo no commit

##### Devemos selecionar uma hash anterior do coomit e aplicar o reset  Soft no hach do segundo commit
$ git reset --soft 06585af5a5e1629f648

Vamos modificar novamente e ve que o segundo commit sumiu e aparece esse novo
$ git commit -m "huehue"

###### Git reset hard apaga tudo, cuidado a usar ele, ele retorna coomit's e ate arquivos que estejam n Stage
git reset --hard 06585af5a5e1629f648d274d2f4a56c102a3816b

#### podemos observar que ele voltou os commits e não tem nada em log
$ git diff
$ git log

##### O git mixed volta o arquivo preparado, mas nao adcionado para stage
git reset --mixed 946aab49b14bb2251ee8b95335dfb7e27733cf1c

dando um git diff podemos vê que ele voltou para o que tinhamos feito e podemos dá um 
checkout, editar, fazer o que quiser

$ git diff

### Ramificação (Branch) 

#### O que é um branch e por que usar? 

###### Branch é um ponteiro móvel que leva o commit vc pode criar para corrigir bug e depois mandar as modificacoes para o repo principal
Quando criamos um repo e cada vez que criamos um coomit ele faz para noix uma hach(conjunto de letras e numeros grande)
e para cada hach ele tira um snapshat(uma foto do estado) o branch faz é apontar para o commit, o primeiro branch
que noix temos ao criar um ropo é o master, e a cada commit ele vai seguindo o commit (foto) começamos no snap A
e estamos no C e o master o acompanha, mas podemos criar um novo branch que aponte para o mesmo coomit,(foto 2) podemos separa
e criar um branch testing e apontando para o mesmo ou podemos ter um branch apontando para um outro commit, (foto diferentes coomit) 
 
    
##### Por que usar,  quais vantagens???
Vantagens:
. posso moddificar meus arquivos sem alterar o local, possso corrigir um bug enquanto outros estao trabalhando no principal
. Facilmente "desligavel" pode apagar e crair com bastante velocidade
. Multiplas pessoas trabalhando. 
. Evita conflhitos. (vc trabalha em um branch sozinho em um amtiente separado, e se todo mundo tah no mesmo branch todo mundo faz commit juto e tem que organizar

 ###### Criando um branch, entrando nele e e listando

###### Entendendo o Merge
Existem duas formas de unir os branchs, o Merge e o Rebase, primeiro vamos explanar o Merge

No merge ele cria um vertice cada um é um ponto, eh uma especie de triangulo, esse circulo juntando esses commits, dessa forma do marge
ele vai criar um coomit extra, juntando os commits separado, com as modificacoes dos branches que estou juntando

###### Pros e contra
Pro, nao eh destrutiva, ele na vai destruir commit nenhum, ele cria um novo coomit 
e nao estraga o historico

Contra: Coomit extra,,, o que nao eh mto legal, e nao tah fazendo nada, mas ele em sim
nao tah fazendo nada, commit nao muito legal

Historico poluido: Quando criamos o manger ele cria a arvore juntando os coomits em forma diamante, fica complicado se tiver muitos branches.

##### Entenddendo o Rebase
O rebase trabalha de forma linear, 
  
Prós: 
Evita commits extras
Historico linear: lina retas
    
Contras:
Perde a ordem cronologica, pega um commit e coloca na frente da fila
Usar ele com o Pull --rebase, assim evita mudança historico
